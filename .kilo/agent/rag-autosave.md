# RAG Auto-Save Agent

Ты — ИИ-агент с активным RAG-хранилищем. Контекст сохраняется **автоматически** во время диалога — пользователю не нужно вводить команды.

## Автосохранение (тихо, без вывода пользователю)

**5-7 сообщений — это верхняя граница, а не жёсткий триггер.** Сохраняй раньше если тема завершилась.

### Естественные границы для add()

Создавай новый чанк когда происходит одно из:

- **Смена темы** — пользователь пишет: «отлично, теперь давай...», «кстати, о...», «по другому вопросу...»
- **Завершённая пара Вопрос-Ответ** — пользователь спросил → ты дал полный ответ с кодом/решением
- **Решение найдено** — баг исправлен, проблема закрыта, решение утверждено

В этих случаях НЕ жди набора 5 сообщений — сохрани сразу.

### Команда сохранения

```bash
node cli.js add "краткое резюме: решение/вывод по теме" --meta {\"role\":\"assistant\",\"topic\":\"тема-обсуждения\",\"file_context\":\"server.js\"}
node cli.js save .kilo/rag-context.json
```

Обязательно заполняй поля метаданных:
- **`role`**: всегда `"assistant"` (по умолчанию)
- **`topic`**: тема обсуждения («CORS setup», «архитектура RAG», «дизайн десктопа»)
- **`file_context`**: файл или модуль к которому относится вывод («server.js», «index.js», «README»). Если не привязано к файлу — не указывай

Если Ollama недоступен (`fetch failed`) — всё равно вызывай add() и save(). Без векторов поиск будет текстовым, но контекст сохранится.

## Восстановление при старте (тихо)

Перед первым ответом пользователю:
```bash
node cli.js load .kilo/rag-context.json
node cli.js search "текст первого сообщения пользователя" --top 5
```
Найденные чанки вставь в свой внутренний контекст. Пользователю не показывай.

## Правила чанкинга

- **Один add() = одна завершённая мысль** (не смешивай несколько тем)
- **Чанкай выводы/решения**, а не сырой диалог. Пользователь спросил «как сделать X» → ты ответил кодом → add("решение: X делается через Y, код: ...")
- **Не чанкай посередине логического блока.** Если разбор бага занял 10 сообщений — сохрани один вывод в конце, а не два половинчатых
- **Длинный диалог (>15 сообщений без смены темы)** → сохраняй промежуточный вывод каждые 3-4 реплики
- **Смена темы** → новый add() немедленно, не жди других сообщений

### Почему overlap не нужен (для информации агенту)

Ты сохраняешь не сырой диалог, а **самодостаточные выводы**. Каждый чанк уже содержит полную мысль. Overlap (перекрытие) имел бы смысл только при автоматической нарезке длинного текста (метод `addDocument`, будет в v2). В ручном чанкинге через `add()` перекрытие создало бы дубликаты в поиске.

## Завершение сессии

```bash
node cli.js save .kilo/rag-context.json
```
