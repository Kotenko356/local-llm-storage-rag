# local-rag-store

Локальное RAG-хранилище. Сохраняет ключевые выводы из диалогов с LLM и находит их потом — даже через неделю, в новом контекстном окне.

**Без БД, без облака, без аренды серверов. Ollama запускается на твоём же компьютере.**

## Зачем это нужно

LLM (ChatGPT, Claude, локальные модели) имеют ограничение контекстного окна — количество токенов, которое модель «помнит» за один раз. Когда диалог становится длинным, старые сообщения выпадают из памяти.

**Без RAG:** агент не помнит, что обсуждалось час назад.

**С RAG:** агент сохраняет ключевые выводы текущего диалога, а при старте новой сессии находит их и подставляет в контекст, как шпаргалку.

---

## Как это работает — простыми словами

Всё происходит на твоём компьютере. Ничего никуда не отправляется.

```
Ты (или агент): ассистент, запомни: "RAG — это поиск + генерация"
                  │
                  ▼
        1. RAG режет текст на куски (чанки)
        2. Отправляет кусок в Ollama (локально, localhost)
        3. Ollama превращает текст в вектор — 
           длинный список чисел, который описывает смысл текста
        4. Вектор сохраняется в оперативной памяти (Map)
                  │
                  ▼   Через час, в новой сессии:

Ты (или агент): найди что-то про RAG
                  │
                  ▼
        1. RAG отправляет твой запрос в Ollama
        2. Ollama тоже превращает запрос в вектор
        3. RAG сравнивает вектор запроса со всеми сохранёнными векторами
        4. Берёт 3-5 самых похожих куска и возвращает как контекст
                  │
                  ▼
        5. Агент подставляет этот контекст в промпт LLM
```

**Персистентность:** векторы хранятся в RAM (быстрый доступ), но при вызове `save()` весь дамп скидывается в JSON-файл на диск. При следующем запуске `load()` восстанавливает всё обратно в RAM.

## Детальный механизм

### Что такое embedding (эмбеддинг)?

Любой текст можно превратить в массив чисел — **вектор**. Длина вектора для `nomic-embed-text` — **768 чисел**. Смысл в том, что у похожих по смыслу текстов получаются похожие векторы (числа близки друг к другу).

### Что такое cosine similarity (косинусное сходство)?

Формула, которая считает «похожесть» двух векторов. Результат — число от 0 до 1:

- **0.95** — тексты почти об одном и том же
- **0.50** — отдалённо похожи
- **0.10** — совсем не связаны

RAG просто перебирает все сохранённые векторы в цикле, считает cosine similarity для каждого и возвращает top-K самых похожих.

### Поиск по 10 000 чанков занимает ~5ms

Никакая БД не нужна. PostgreSQL + pgvector дали бы 800 µs (чуть быстрее), но установка и конфигурация не стоят того до 50 000+ чанков.

### Что происходит с текстом при add()

```
add("RAG — это поиск + генерация", { role: "assistant" })
        │
        ├─→ role = "assistant" (по умолчанию — assistant)
        ├─→ conversationId = "k3f9x2..." (авто, если не указан)
        ├─→ timestamp = Date.now()
        ├─→ messageIndex = 0, 1, 2... (автоинкремент внутри conversationId)
        ├─→ id = "chunk_m1x3k9" (уникальный идентификатор)
        │
        ├─→ embedder.js → Ollama POST /api/embed
        │                    ↓
        │              [0.12, -0.34, 0.56, ...]  (768 чисел)
        │
        └─→ chunks.set(id, { v: [...], text: "RAG — это...", meta: {...} })
```

## Требования

- **Node.js 18+** — среда выполнения JavaScript. Проверить: `node --version`
- **Ollama** — программа для запуска LLM-моделей локально. Бесплатно, без регистрации.

### Почему именно Ollama?

Потому что Ollama запускается локально: твой процессор, твоя память, без интернета, без API-ключей, без лимитов запросов. Альтернативы (OpenAI Embeddings, Cohere) — платные и отправляют текст на сервер.

### Почему nomic-embed-text?

| Модель | Размер | Качество | Запуск |
|--------|--------|----------|--------|
| `nomic-embed-text` | 137 MB | Хорошее | Локально, бесплатно |
| OpenAI `text-embedding-3-small` | — | Отличное | Через API, платно |
| `mxbai-embed-large` | 334 MB | Лучше | Тоже Ollama, больше RAM |

Для RAG-хранилища качества `nomic-embed-text` достаточно, а размер минимальный. Если нужно лучше — можно указать другую модель в конструкторе.

## Установка Ollama — подробно

### Шаг 1: Скачать и установить

1. Перейти на [ollama.com/download](https://ollama.com/download)
2. Скачать под свою ОС (Windows, macOS, Linux)
3. Установить как обычную программу (для Windows — установщик `.exe`)

После установки Ollama автоматически запускается как фоновый сервис. Проверить:

```powershell
ollama --version
```

### Шаг 2: Скачать модель эмбеддинга

```powershell
ollama pull nomic-embed-text
```

Скачивание ~137 MB. Выполняется один раз.

### Шаг 3: Убедиться, что сервер запущен

Ollama слушает порт `localhost:11434`. Проверить:

```powershell
curl.exe http://localhost:11434/api/tags
```

Если ответ — JSON со списком моделей, всё работает. Если ошибка — запустить `ollama serve` (или перезапустить Ollama из системного трея).

### Сколько это ест ресурсов

| Параметр | Значение |
|----------|----------|
| Размер модели на диске | 137 MB |
| RAM в простое | ~150–300 MB (модель загружена в память) |
| RAM на один запрос | не растёт (единоразовая загрузка) |
| CPU в простое | 0% |
| CPU при эмбеддинге | ~10–15% одного ядра |
| Время на один эмбеддинг | 5–10 ms (текст длиной 500 символов) |
| Сеть | Только localhost, ничего не уходит в интернет |

Для сравнения: одна вкладка Chrome с Gmail ест больше RAM, чем nomic-embed-text.

## API

Импорт и создание экземпляра:

```js
import { LocalRAG } from './index.js';

const rag = new LocalRAG({
  model: 'nomic-embed-text'   // какая модель Ollama используется
});
```

### `add(text, meta)` → `id`

Сохраняет текст в хранилище. Сначала превращает текст в вектор (через Ollama), потом сохраняет в Map.

| Параметр meta | Обязательный | По умолчанию | Описание |
|---------------|-------------|-------------|----------|
| `role` | да (всегда есть) | `'assistant'` | Кто сказал: user / assistant / system |
| `conversationId` | нет | авто-uuid | Группировка сообщений в один диалог |
| `timestamp` | нет | `Date.now()` | Когда было сказано |
| `messageIndex` | нет | автоинкремент | Порядковый номер в диалоге |
| любые свои поля | нет | — | Сохраняются как есть в `meta` |

```js
const id = await rag.add('RAG расшифровывается как Retrieval-Augmented Generation', {
  role: 'assistant',
});

const id2 = await rag.add('а что такое fine-tuning?', {
  role: 'user',
  conversationId: 'session-may-17',
  topic: 'ml-basics',
});
```

### `search(query, topK)` → `[{ text, similarity, id, meta }, ...]`

Ищет `topK` самых похожих на `query` чанков. Тоже эмбеддит запрос через Ollama.

```js
const results = await rag.search('как работает RAG', 3);
// [
//   { text: 'RAG расшифровывается как...', similarity: 0.94, id: 'chunk_x3f1', meta: {...} },
//   { text: 'RAG — это поиск + генерация', similarity: 0.88, id: 'chunk_z9k2', meta: {...} },
// ]
```

### `remove(id)` → `boolean`

Удаляет один чанк по его id. Возвращает `true`, если чанк был найден.

```js
rag.remove('chunk_x3f1');
```

### `removeByConversation(conversationId)` → `number`

Удаляет все чанки, принадлежащие указанному диалогу. Возвращает количество удалённых.

```js
rag.removeByConversation('session-may-17');
```

### `clear()`

Очищает всё хранилище (и RAM, и счётчики).

### `save(path)`

Сохраняет всё содержимое RAM в JSON-файл на диск.

```js
await rag.save('./my-rag-dump.json');
```

### `load(path)`

Загружает дамп с диска обратно в RAM. Восстанавливает все векторы, текст, метаданные и счётчики сообщений.

```js
await rag.load('./my-rag-dump.json');
```

### `size()` → `number`

Количество чанков в хранилище.

## CLI

Все команды вызываются из PowerShell / bash. **Каждая команда возвращает JSON в stdout** — агент (KiloCode) может распарсить результат.

```powershell
# Добавить чанк. role = "assistant" по умолчанию (агент сохраняет свои выводы)
node cli.js add "RAG — это поиск + генерация. Сначала ищем релевантные чанки, потом докидываем их в промпт LLM"

# Явно указать роль user (если это реплика пользователя)
node cli.js add "спросил про разницу между RAG и fine-tuning" --meta "{\"role\":\"user\"}"

# Поиск top-5 релевантных чанков
node cli.js search "как работает RAG" --top 5

# Сохранить дамп на диск (файл создаётся в текущей папке)
node cli.js save .kilo/context-dump.json

# Загрузить дамп с диска
node cli.js load .kilo/context-dump.json

# Очистить всё (безвозвратно, если не сохранено)
node cli.js clear

# Удалить один чанк по id
node cli.js remove chunk_m1x3k9

# Удалить все чанки диалога
node cli.js remove-by-conversation session-may-17
```

### Пример: результат search

```json
{
  "results": [
    {
      "text": "RAG расшифровывается как Retrieval-Augmented Generation",
      "similarity": 0.94,
      "id": "chunk_x3f1",
      "meta": { "role": "assistant", "conversationId": "abc123", "timestamp": 1715930000000, "messageIndex": 0 }
    }
  ]
}
```

## Интеграция с KiloCode

KiloCode-агент вызывает CLI через bash. В начале сессии загружает старый контекст и ищет релевантное к текущей задаче. В конце — сохраняет новые выводы.

```bash
# Старт сессии — если есть дамп, восстанавливаем и ищем
if [ -f .kilo/rag-context.json ]; then
  node cli.js load .kilo/rag-context.json
  node cli.js search "текущая задача из сообщения пользователя" --top 5
fi

# В ходе сессии — add по ключевым выводам

# Конец сессии
node cli.js save .kilo/rag-context.json
```

Дамп хранится в `.kilo/rag-context.json` — внутри проекта, рядом с сессиями KiloCode.

## Ollama: ресурсы, порты, запуск

### Проверка статуса

```powershell
# Проверить, запущен ли Ollama
curl.exe http://localhost:11434/api/tags
# Должен вернуть JSON {"models": [...]}

# Посмотреть какие модели загружены
ollama list

# Показать детали модели
ollama show nomic-embed-text
```

### Если сервер недоступен

`add()` и `search()` вернут ошибку `fetch failed`, потому что некому превратить текст в вектор. Команды, которым не нужны векторы, работают без Ollama:

| Команда | Работает без Ollama? |
|---------|---------------------|
| `add` | ❌ Нужен эмбеддинг |
| `search` | ❌ Нужен эмбеддинг запроса |
| `save` | ✅ Просто запись JSON |
| `load` | ✅ Просто чтение JSON |
| `remove` | ✅ Манипуляция Map |
| `remove-by-conversation` | ✅ Манипуляция Map |
| `clear` | ✅ Очистка Map |

### Разные модели

```js
// Можно использовать любую модель Ollama, которая поддерживает эмбеддинги
const rag = new LocalRAG({ model: 'mxbai-embed-large' });
```

## Дополнительные ответы на вопросы

### Нужно ли арендовать сервер?

Нет. Всё работает на твоём компьютере. Ollama — это обычная программа, как браузер или VS Code. Она слушает `localhost:11434` — только локальные запросы, ничего не уходит в интернет.

### Данные уходят в интернет?

Нет. Весь трафик — между RAG-хранилищем и Ollama через `localhost`. Никакие тексты, векторы или запросы не покидают твой компьютер.

### Что будет, если закрыть терминал?

Всё, что сохранено через `save()`, лежит в JSON-файле на диске и никуда не денется. Данные в RAM (то, что не сохранено) пропадут — поэтому перед завершением сессии нужно вызывать `save()`.

### Можно перенести дамп на другой компьютер?

Да. JSON-файл — обычный текст. Скопировал на другую машину, выполнил `load()` — и контекст доступен. На другой машине тоже должен быть запущен Ollama с той же моделью.

### Почему не используется БД (PostgreSQL, SQLite)?

Для объёмов до 50 000 чанков (2 GB) Map в RAM + цикл по векторам занимает ~5ms на поиск. БД даёт выигрыш только на сотнях тысяч векторов. Для RAG-хранилища из диалогов с LLM 50 000 чанков — это годы общения. БД была бы оверхедом.

### Чем отличается save/load от add/search?

- **add/search** — текущая сессия: добавить в RAM, найти в RAM
- **save/load** — персистентность: сбросить RAM на диск, восстановить с диска в RAM

### Что такое Chunk ID? Где его взять?

`add()` возвращает id строкой вида `chunk_m1x3k9`. Можно сохранить его в переменной. Также id можно увидеть в результатах `search()` — каждое совпадение содержит `id`. Если id потерян, но известен `conversationId`, можно удалить весь диалог через `removeByConversation()`.

### А если я хочу удалить только одно сообщение посреди диалога?

Понадобится `conversationId` и метод `remove()` + знание id. На практике проще: `clear()` → передобавить нужные сообщения → `save()`.

### Можно автоматически зачанить длинный документ?

Сейчас нет. `add()` принимает только готовые куски текста. Агент сам решает, что и как чанкать. Автоматическое чанкование документа (`addDocument`) — возможное будущее расширение.

### На каких ОС работает?

Windows, macOS, Linux. Ollama и Node.js кроссплатформенные.
